Зачем нужна балансировка

Когда один сервер перестаёт справляться с нагрузкой, трафик распределяют между несколькими одинаковыми серверами. Балансировщик стоит перед ними и решает, на какой сервер отправить каждый входящий запрос. Алгоритм этого решения и определяет, насколько равномерно ляжет нагрузка и не перегрузится ли один из серверов, пока другие простаивают.

Главный вопрос при выборе алгоритма: запросы одинаковые по времени выполнения или разные? От этого зависит почти всё. Для коротких однотипных запросов хватает простого Round-Robin; для долгих и разных по тяжести нужен алгоритм, который следит за реальной загрузкой серверов.

01

Round-Robin

Round-Robin

Запросы распределяются строго по кругу

простой
server = (current_index++) % server_count

Каждый новый запрос идёт на следующий сервер по очереди: 1 → 2 → 3 → 1 → 2 → 3. Никакой логики, просто очередь. Работает идеально, когда все запросы одинаковые по весу, а серверы одинаковые по мощности.

Как это работает

Суть: балансировщик хранит лишь указатель на следующий сервер и сдвигает его по кругу — решение за O(1). Реальную загрузку серверов он при этом не видит.

02

Weighted Round-Robin

Weighted Round-Robin

Round-Robin, но с весами для каждого сервера

простой
сервер с весом 3 получает 3 запроса на каждый 1 у сервера с весом 1

Каждому серверу назначается вес, обычно пропорционально мощности. Сервер с весом 3 получит втрое больше трафика. Используют, когда железо разное: один сервер на 64 ГБ RAM, другой на 16 ГБ.

Как это работает

Суть: толщина стрелки = доля трафика. Веса задают вручную, обычно пропорционально мощности сервера.

подходит

03

Least Connections

Least Connections

Запрос идёт на сервер с наименьшим числом активных соединений

умный
server = argmin(active_connections[i])

Балансировщик постоянно следит, сколько активных соединений у каждого сервера. Новый запрос идёт туда, где их меньше всего. Если сервер завис с тяжёлыми долгими запросами — новые к нему не пойдут.

Как это работает

Суть: балансировщик опрашивает счётчики активных соединений и направляет запрос туда, где их меньше всего.

04

Least Response Time

Least Response Time

Учитывает и нагрузку, и скорость ответа сервера

продвинутый
score = active_connections × avg_response_time → выбираем min(score)

Самый умный из распространённых алгоритмов. Балансировщик смотрит на два параметра сразу: число активных соединений и среднее время ответа. Быстрый, но уже загруженный сервер может проиграть медленному, но свободному. Встречается в HAProxy, Envoy, NGINX Plus.

Как это работает

Суть: учитываются сразу два фактора — загрузка и скорость. Быстрый, но занятый сервер может проиграть медленному, но свободному.

05

IP Hash

IP Hash (Sticky Sessions)

Один клиент — всегда один и тот же сервер

сессионный
server = hash(client_ip) % server_count

IP-адрес клиента хэшируется и определяет сервер. Один пользователь всегда попадает на один и тот же сервер. Нужно, когда состояние сессии хранится локально на сервере — например, файл загружается частями или сессия лежит в памяти, а не в Redis.

Как это работает

Суть: функция детерминирована — один и тот же IP всегда даёт один сервер. Платой становится перекос: за общим NAT-адресом весь трафик уйдёт на один сервер.

Random

Random

Случайный выбор сервера

простой
server = random(0, server_count - 1)

Сервер выбирается случайно. Звучит примитивно, но при большом числе запросов (закон больших чисел) распределение получается практически равномерным — как Round-Robin, но без хранения счётчика. Используется в некоторых P2P- и DNS-балансировщиках.

Как это работает

Суть: выбор случайный и без состояния. По закону больших чисел при большом потоке доли серверов выравниваются.

подходит

не подходит

Как выбирать на интервью

Если спросят, какой алгоритм использовать, сначала задайте встречный вопрос: запросы одинаковые или разные по времени выполнения? Дальше выбор почти очевиден.

Запросы короткие и одинаковые
→Round-Robin
Запросы разные по тяжести
→Least Connections
Серверы разной мощности
→Weighted Round-Robin
Нужны sticky sessions
→IP Hash
Максимальная точность на highload
→Least Response Time

В реальных системах часто комбинируют. Weighted Least Connections — это Least Connections, но с весами. Именно его HAProxy использует по умолчанию для HTTP-трафика: учитывает и вес сервера, и его текущую загрузку.
